# 战斗设计规则

本项目面向 Cocos Creator 3.8.7 + TypeScript，并会发布到微信、抖音等小游戏正式环境。战斗逻辑通常处于高频更新路径，设计时要优先考虑组合表达、接口稳定、低分配和可退化实现。

## 触发范围

涉及以下任一主题时读取本文：

- 战斗实体、角色、怪物、召唤物、子弹、碰撞体、场景物件。
- 技能、Buff、伤害、命中、状态机、AI 行为、战斗事件。
- 战斗数值流、战斗表现同步、战斗回放、战斗模拟。
- 任何 battle / combat / fight 相关架构或高频逻辑设计。

## 核心原则

1. 组合大于继承。战斗对象的能力和状态优先用数据组件、标签组件、系统处理、配置数据和明确的组合关系表达。
2. 不用深继承树表达玩法变化。少量稳定基类可以保留，但不要把角色类型、技能类型、Buff 类型、伤害类型一路继承展开。
3. 运行时接口要克制。小游戏正式环境会转成底层 JavaScript 运行，接口层级、动态派发、反射式查找和运行时生成都要谨慎。
4. 高频路径少分配。`update`、命中检测、Buff tick、技能结算等逻辑避免每帧创建临时对象、闭包、数组和字符串 key。
5. 数据和表现分离。战斗结算优先保持纯数据流程，Cocos 节点、动画、特效、音效作为表现层或可替换组件接入。

## 现有 ECS 评估

Tyou 已有 `tyou.ecs`，源码位于 `Client/assets/ty-framework/core/ecs/`，并在 `Tyou.onUpdate(dt)` 中执行。

从当前源码看，它适合优先评估：

- `ECSComp` 要求组件主要承载数据，并通过 `reset()` 支持复用。
- `ECSEntity` 通过添加/移除组件表达实体能力，组件和实体都有缓存池。
- `ECSMatcher` / `ECSMask` 使用组件 id 和 `Uint32Array` 位掩码做匹配。
- `ECSGroup` 缓存匹配实体列表，在组件变化时失效重建。
- `ECSRootSystem` 会把组合系统摊平到根系统执行，减少嵌套调用。

因此，战斗设计时先判断现有 ECS 是否满足需求：

- 若需求是实体 + 组件 + 系统的常规组合模型，且性能压力可控，优先直接使用 `tyou.ecs`。
- 若只需要很小的局部状态流，不必强行套 ECS，可用业务侧轻量数据结构。
- 若现有 ECS 的匹配、生命周期、调试、回放、确定性或性能模型不适合，允许从 0 设计战斗专用轻量方案。
- 放弃现有 ECS 时必须说明原因，不要默认重造；使用现有 ECS 时也不要修改 `Client/assets/ty-framework/`，除非开发者明确确认框架扩展。

## 小游戏运行时约束

微信、抖音等小游戏正式环境对包体、启动、GC 和 JavaScript 动态特性更敏感。战斗接口设计时遵循：

- 避免大而全的抽象接口和多层继承，只暴露少量稳定入口。
- 避免依赖类名、运行时反射、动态扫描、字符串拼接查方法。
- 避免运行时生成代码、动态 import 大块逻辑或频繁创建函数对象。
- 避免在高频路径使用过多泛型包装、Promise 链、事件广播和临时容器。
- 对跨系统通信优先使用明确数据结构、组件状态或队列，避免无限制全局事件。
- 资源、特效和 UI 仍遵循现有 `tyou.res`、`UIBase`、引用计数和自动释放规则。

## AI 输出要求

当 AI 设计战斗相关方案时，必须说明：

1. 哪些变化用组合表达，哪些少量基类是稳定基础。
2. 是否评估过 `tyou.ecs`，结论是复用、局部复用还是不采用。
3. 若不采用现有 ECS，为什么它不适合当前需求，以及轻量方案如何控制复杂度。
4. 高频路径如何避免每帧分配、深层动态派发和小游戏运行时不友好的接口。
5. 是否触及 `Client/assets/ty-framework/`；若触及，先暂停并请求开发者确认。
