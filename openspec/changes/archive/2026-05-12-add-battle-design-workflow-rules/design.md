## Context

Tyou 的运行时已经暴露 `tyou.ecs`，并在主循环中执行 `ecs.execute(dt)`。现有 ECS 源码体现了组件数据、实体组合、系统处理、组件池、实体池、位掩码匹配和系统摊平执行等设计，适合作为战斗系统建模时的优先评估对象。

同时，本项目是 Cocos Creator 面向小游戏环境的客户端框架，正式发布到微信、抖音等小游戏平台后会转换为底层 JavaScript 运行。战斗系统通常处在高频更新路径，AI 生成方案必须避免只追求抽象完整而忽视运行时成本。

## Goals / Non-Goals

**Goals:**

- 明确战斗设计遵循组合大于继承。
- 明确 AI 需要先评估现有 ECS 是否适合当前战斗需求。
- 明确小游戏正式环境下的接口、抽象和高频分配约束。
- 将规则放入现有 AI 工作流结构：入口短规则、`tyou-dev` 路由、细节 reference、人读 workflow、OpenSpec spec。

**Non-Goals:**

- 不修改 `Client/assets/ty-framework/` 中的 ECS 实现。
- 不强制所有战斗系统必须使用现有 ECS。
- 不引入新的战斗框架、第三方 ECS 或运行时依赖。
- 不设计具体战斗玩法、技能、Buff 或数值系统。

## Decisions

1. 战斗设计规则放入新的 `battle-design.md` reference。
   - 原因：战斗设计是独立主题，细节不应塞进 `AGENTS.md` 或 `SKILL.md`，否则会增加所有任务的上下文噪声。
   - 替代方案：直接写入 `workflow-recovery.md`。不采用，因为该文档负责容错与文档同步，不负责业务设计原则。

2. 入口文档只保留短规则。
   - 原因：`AGENTS.md` 是每次会话入口，应只保存必须知道的红线。
   - 替代方案：在入口展开完整 ECS 说明。会增加非战斗任务负担，不采用。

3. 对现有 ECS 采用“优先评估，合适复用，不合适从 0 设计轻量方案”。
   - 原因：ECS 源码已有对象池、位掩码、系统摊平等性能倾向，也符合组合优先；但它位于受保护框架目录，且当前没有业务战斗样例，不能写成绝对强制。
   - 替代方案：强制使用 `tyou.ecs`。不采用，避免把所有战斗需求绑定到一个可能不适合具体玩法的抽象。

4. 明确小游戏 JS 运行时约束。
   - 原因：战斗代码处于高频路径，微信/抖音等小游戏正式环境对包体、启动、GC、动态特性和接口稳定性更敏感。
   - 替代方案：只写“注意性能”。不采用，因为过于宽泛，不能指导 AI 设计接口。

## Risks / Trade-offs

- 规则过强导致 AI 回避合理继承 → 明确允许少量稳定基类，但禁止用深继承表达玩法变化。
- 规则过弱导致 AI 每次重造 ECS → 要求先评估现有 ECS，并说明复用或放弃原因。
- 新 reference 被遗漏 → 在 `tyou-dev/SKILL.md` 文档路由和 `Books/AI-Development-Workflow.md` 当前能力中登记。
- OpenSpec delta 未同步到主 spec → 实施后同步 `openspec/specs/codex-ai-workflow/spec.md`。
